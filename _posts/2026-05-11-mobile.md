## Mobile Operating Systems

* Android OS (Google Inc.)
  * A software platform and operating system for mobile devices.
  * Based on the Linux kernel.
    * Android uses Linux for its device drivers, memory management, process management, and networking.
    * The next level up contains the Android native libraries. They are all written in C / C++ internally, but you'll be calling them through Java / Kotlin interfaces. In this layer you can find the Surface Manager, 2D and 3D graphics, media codecs, the SQL database (SQLite), and a native web browser engine (WebKit).
  * Developed by Google and later the Open Handset Alliance (OHA).
  * Dalvik Virtual Machine & Android Runtime (ART) Dalvik 虚拟机 & Android 运行时 (ART)
    * Managed runtime used by applications and some system services on Android. Android 上供应用程序和某些系统服务使用的托管运行时。
    * Android 4.4 (KitKat) brought a technology preview of ART, including it as an alternative runtime environment and keeping Dalvik as the default virtual machine. In the subsequent major Android release, Android 5.0 (Lollipop), Dalvik was entirely replaced by ART. Android 4.4 (KitKat) 引入了 ART 的技术预览版，将其作为一种替代运行时环境，并保留 Dalvik 作为默认虚拟机。在随后的主要 Android 版本 Android 5.0 (Lollipop) 中，Dalvik 被 ART 完全取代。
    * Key difference:
      * Dalvik: interpretation and trace-based JIT compilation Dalvik：解释与基于跟踪的即时编译
      * Android Runtime (ART): Android运行时（ART）：
        * ahead-of-time (AOT) compilation, compiling entire applications into native machine code upon their installation -> improves the overall execution efficiency and reduces power consumption, which results in improved battery autonomy on mobile devices 提前（AOT）编译，在安装时将整个应用程序编译成本地机器代码，可提高整体执行效率并降低功耗，从而提高移动设备的电池续航能力
        * faster execution of applications, improved memory allocation and garbage collection (GC) mechanisms, new applications debugging features, and more accurate high-level profiling of applications 更快的应用程序执行速度、改进的内存分配和垃圾回收（GC）机制、新的应用程序调试功能以及更准确的应用程序高级性能剖析
  * Security: 安全性：
    * Android is a multi-process system, in which each application (and parts of the system) runs in its own process. Most security between applications and the system is enforced at the process level through standard Linux facilities, such as user and group IDs that are assigned to applications. Android 是一个多进程系统，其中每个应用程序（以及系统的一部分）都在自己的进程中运行。应用程序与系统之间的大部分安全都通过标准的 Linux 机制在进程级别强制执行，例如分配给应用程序的用户和组 ID。
    * Additional finer-grained security features are provided through a "permission" mechanism that enforces restrictions on the specific operations that a particular process can perform, and per-URI permissions for granting ad-hoc access to specific pieces of data. 通过"权限"机制提供了额外的更细粒度的安全功能，该机制强制执行对特定进程可以执行的特定操作的限制，以及用于授予对特定数据片段的临时访问的每个 URI 的权限。
  * Programming languages: 编程语言：
    * Java / Kotlin – officially supported
    * C / C++ – as native library that Gradle can package with APK (using build tool CMake), and being called by Java / Kotlin program via Java Native Interface (JNI) C / C++ – 作为 Gradle 可以随 APK 一起打包的原生库（使用构建工具 CMake），并通过 Java Native Interface (JNI) 由 Java / Kotlin 程序调用

## Rules for Android UI

* Do not block the UI thread
  * *Rationale*: The UI thread is the single thread responsible for drawing the interface, handling touch events, and maintaining responsiveness. If it is occupied with a long‑running operation (network call, file I/O, complex calculation), it cannot process user input or redraw the screen. This causes the app to freeze, resulting in an "Application Not Responding" (ANR) dialog and a severely degraded user experience. Therefore, all potentially slow or blocking work must be moved to background threads.
* Do not access the UI toolkit from outside the UI thread
  * *Rationale*: The Android UI toolkit (`View`, `TextView`, `Button`, etc.) is not designed with thread‑safety mechanisms for concurrent access. If a background thread tries to read or modify a UI element, it can introduce race conditions, leave the view hierarchy in an inconsistent state, or cause the system to throw a `CalledFromWrongThreadException`. To guarantee a deterministic, crash‑free UI, the toolkit is restricted to the thread that created it (the main UI thread). Any UI update originating from another thread must be explicitly dispatched back to the UI thread using mechanisms like `View.post()`, `Handler`, or `runOnUiThread()`.


## Difference between thread and handler

* **Thread**: A unit of execution that runs code in parallel to the main thread. Used to perform background tasks.
* **Handler**: An object tied to a thread's `Looper` and message queue. It is used to post `Runnables` or messages to that specific thread, enabling safe communication (e.g., from background to UI thread). A handler is not a thread itself.

### Function

| Function Definition | Function Call |
| :-----------------: | :-----------: |
| **With both argument labels and parameter names:** `func ABC(e1 i1: Int, e2 i2: Int, e3 i3: Int)` | `ABC(e1:75, e2:6, e3:2020)` |
| **With only parameter names:** `func ABC(i1: Int, i2: Int, i3: Int)` | `ABC(i1:75, i2:6, i3:2020)` |
| **Without argument labels and parameter names:** `func ABC(_ i1: Int, _ i2: Int)` | `ABC(75, 6)` |

* You can define a default value for any parameter in a function by assigning a value to the parameter after that parameter's type. If a default value is defined, you can omit that parameter when calling the function.
* **Variadic Parameters**: A variadic parameter accepts zero or more values of a specified type. You use a variadic parameter to specify that the parameter can be passed a varying number of input values when the function is called. Write variadic parameters by inserting three period characters (...) after the parameter's type name. A function may have at most one variadic parameter.
* **In-Out Parameters**: Function parameters are constants by default. Trying to change the value of a function parameter from within the body of that function results in a compile-time error. This means that you can't change the value of a parameter by mistake. If you want a function to modify a parameter's value, and you want those changes to persist after the function call has ended, define that parameter as an in-out parameter instead.

```java
// MainActivity.java
package hk.hkucs.comp7705timer; 
import android.support.v7.app.AppCompatActivity; 
import android.os.*; 
import android.widget.*; 
import android.view.View; 
import android.media.MediaPlayer; 

public class MainActivity extends AppCompatActivity { 
  static int maxTime = 1800; 
  int time; 
  Button startButton, stopButton, resetButton; 
  TextView timeDisplay; 
  Boolean timerEnabled; 
  private Handler handler = new Handler(); 

  @Override 
  protected void onCreate(Bundle savedInstanceState) { 
    super.onCreate(savedInstanceState); 
    setContentView(R.layout.activity_main); 
    time = maxTime; 
    timerEnabled = true; 
    startButton = (Button) findViewById(R.id.startButton); 
    stopButton = (Button) findViewById(R.id.stopButton); 
    resetButton = (Button) findViewById(R.id.resetButton); 
    timeDisplay = (TextView) findViewById(R.id.timeDisplay);

    int Minutes = time / 60; 
    int Seconds = time % 60; 
    timeDisplay.setText(Minutes + ":" + Seconds); 

    MediaPlayer mediaPlayer; 
    mediaPlayer = MediaPlayer.create(this, R.raw.kpsong); 
  }

  public void startAction(View view) { 
    handler.removeCallbacks(updateTimer); 
    handler.postDelayed(updateTimer, 1000); 
    timerEnabled = true; 
  }

  public void stopAction(View view) { 
    timerEnabled = false; 
  }

  public void resetAction(View view) { 
    time = maxTime; 
    int Minutes = time / 60; 
    int Seconds = time % 60; 
    timeDisplay.setText(Minutes + ":" + Seconds); 
    mediaPlayer.stop(); 
  }

  private Runnable updateTimer = new Runnable() { 
    public void run() { 
      time = time - 1; 
      int Minutes = time / 60; 
      int Seconds = time % 60; 
      timeDisplay.setText(Minutes + ":" + Seconds); 
      if ((timerEnabled) && (time > 0)) { 
        handler.postDelayed(this, 500); 
      } else { 
        mediaPlayer.start(); 
      }
    }
  };
}
```

1. The XML attributes `android:onClick="startAction"` etc. tell Android to call the named method in the attached Activity when the button is clicked. The method must be `public`, return `void`, and take a `View` parameter. No programmatic listener registration is needed.
2. `maxTime = 1800` seconds = 30 minutes. The `Runnable` decrements `time` by **1 every 500 ms** (`handler.postDelayed(this, 500)`). Therefore each real second reduces the counter by 2, so the timer runs out after **15 minutes** (900 seconds). **Fix**: Change the `post‑delay` inside `updateTimer` from `500` to `1000` milliseconds. Also synchronise the initial delay in `startAction` to 1000 (already set) so that the tick rate is consistent – now `time` counts true seconds.
3. Two rules for Android UI control and rationale
  * **Do not block the UI thread**: The UI thread is the single thread responsible for drawing the interface, handling touch events, and maintaining responsiveness. If it is occupied with a long‑running operation (network call, file I/O, complex calculation), it cannot process user input or redraw the screen. This causes the app to freeze, resulting in an "Application Not Responding" (ANR) dialog and a severely degraded user experience. Therefore, all potentially slow or blocking work must be moved to background threads.
  * **Do not access the UI toolkit from outside the UI thread**: The Android UI toolkit (`View`, `TextView`, `Button`, etc.) is not designed with thread‑safety mechanisms for concurrent access. If a background thread tries to read or modify a UI element, it can introduce race conditions, leave the view hierarchy in an inconsistent state, or cause the system to throw a `CalledFromWrongThreadException`. To guarantee a deterministic, crash‑free UI, the toolkit is restricted to the thread that created it (the main UI thread). Any UI update originating from another thread must be explicitly dispatched back to the UI thread using mechanisms like `View.post()`, `Handler`, or `runOnUiThread()`.
4. Thread vs Handler; can a thread implement this timer?
  * **Thread**: A unit of execution that runs code in parallel to the main thread. Used to perform background tasks.
  * **Handler**: An object tied to a thread's `Looper` and message queue. It is used to post `Runnables` or messages to that specific thread, enabling safe communication (e.g., from background to UI thread). A handler is not a thread itself.
  * No. UI updates must be executed on the main thread.

  1. Four principles of object-oriented programming.
    * Abstraction: The act of representing essential features without including background details.
    * Encapsulation: The act of wrapping up of data and operations into a single unit.
    * Inheritance: The process by which objects of one class acquire the properties of objects of another class.
    * Polymorphism: The ability to take more than one form.
  2. Why OOP is important for mobile app development.
    * Mobile frameworks (Android's Activity/Fragment, iOS's UIViewController) are inherently object‑oriented. OOP enables code reuse, modularity, and maintainability. It allows developers to extend base components, encapsulate UI logic, and manage complex, stateful interfaces cleanly.
  3. One advantage and one disadvantage of Android & iOS development.
    * Android
    * Advantage: Larger global market share; more hardware diversity.
    * Disadvantage: Fragmentation – testing across many device sizes, OS versions, and custom ROMs is complex.
    * iOS
    * Advantage: Uniform ecosystem; easier to optimise and test, higher per‑user revenue.
    * Disadvantage: Strict App Store review guidelines; less freedom in system‑level customisation.
  4. Rationale for React Native/Flutter: reach more customers quickly (performance not optimal).
    * Cross‑platform frameworks allow a single codebase to target both iOS and Android. This drastically reduces development time and cost, enabling faster time‑to‑market and the ability to "reach more customers quickly." For many business applications, the small performance trade‑off is acceptable compared to the huge gain in speed and resource savings.
  5. What is hybrid app? Which has higher cost & better performance (hybrid/native)?
    * A hybrid app is a web application (HTML/CSS/JS) wrapped inside a native container (e.g., Cordova), rendered through a WebView.
    * Native development incurs higher cost (separate codebases) but delivers better performance and full access to device APIs. Hybrid apps have lower initial cost but poorer performance and UX.
  6. What are penetration testing & stress testing? Why important for government apps?
    * **Penetration testing**: authorised simulated attack on the app to identify security vulnerabilities.
    * **Stress testing**: evaluating the app's behaviour under extreme load (many users, low memory, network flood).
    * Importance for government apps: they handle sensitive citizen data and must be highly available. Penetration tests protect against data breaches; stress tests ensure the app remains stable and responsive during high‑demand periods, safeguarding public trust and continuity of service.
  7. What are argument label and parameter name in Swift function definition?
    * **Argument label**: used at the call site to make the function call expressive (e.g., func greet(person name: String) – greet(person: "Alice")).
    * **Parameter name**: used inside the function body to refer to the argument's value (e.g., name). If no label is given, the parameter name serves as both.

  11. Both Android Studio and Xcode allow users to sign an application for running on real mobile devices. Please state the key differences in the signature process with Android Studio and Xcode.
    * **Android**: Uses a self-generated Java Keystore (`.jks`/`.keystore`). Developers can self-sign their APK/AAB. Google Play App Signing optionally manages release keys.
    * **iOS**: Requires an Apple-issued certificate tied to a paid developer account, plus a provisioning profile that specifies allowed devices and capabilities. Signing is managed by Xcode with an Apple ID — self-signing is not allowed for distribution.

  13. To publish to Google Play Store, we need to provide `versionCode` and `versionName` for our application. What is the difference between them?

  15. What is the difference between hardware sensor and software sensor on a mobile device? Please give one example for each of them.
    * **Hardware sensor**: A physical component built into the device that directly measures environmental properties. Example: accelerometer.
    * **Software sensor**: A virtual sensor that derives data by combining and processing data from one or more hardware sensors. Example: rotation vector sensor.
