## 面试题

### Agent

1. Agent设计里最重要的部分是什么
  * Agent设计里最重要的部分是**决策-执行-反馈**闭环的可靠性与可扩展性，具体体现在：1.规划与推理：模型能否根据目标拆解任务、制定计划、调整策略。2.工具使用：能否准确选择并调用外部工具，处理工具返回结果。3.记忆与状态管理：在多轮交互和长任务中保持上下文一致，记忆相关历史。4.错误恢复与安全边界：遇到错误时能否自我纠正，避免死循环或危险操作。5.可控性：人类可干预、可中止，行为可解释、可审计。
  * 若只挑一个最核心的，则是**规划与工具使用的结合**——即模型如何把模糊意图转化为具体、可执行的动作序列，并在环境反馈中动态调整。
2. 主流的Agent设计
  * 主流的Agent架构模式包括：1.**ReAct**：交替进行推理（Thought）和动作（Action），将推理痕迹与观察交织，适合需要逐步推理的场景。2.**Plan-and-Execute**：先生成完整计划，再逐步执行，执行过程中可根据反馈重规划。3.**LLM Compiler / Parallel Tool Use**：并行调用多个工具，合并结果，提升效率。4.**Hierarchical Agent**：主Agent将任务分派给多个子Agent，每个子Agent专注于特定领域或工具。5.**Reflexion / Self-Refine**：在执行后自我评价并修正，适用于代码生成、写作等任务。6.**Code-as-Action**：让模型直接生成可执行代码（如Python）来完成任务，利用代码解释器执行复杂逻辑。工程化框架（LangChain、AutoGPT、CrewAI等）多数围绕这些模式构建。
3. 怎么评估一个Agent系统的效果
  * 评估Agent系统需要多维度的指标：1.任务成功率：端到端完成指定任务的比例。2.工具选择准确率：是否在正确时机调用正确工具，参数是否正确。3.执行效率：步数、时间、Token消耗。4.鲁棒性：面对干扰、工具故障或意外输入时的表现。5.安全性：是否产生危险动作，有无越权访问。6.可解释性：推理链是否清晰、可审计。7.用户满意度：人工评估或偏好评分。
  * 评测方式包括：1.基准测试集（如WebArena、GAIA、SWE-bench、ToolBench等）。2.仿真环境（模拟操作系统、网页、API）。3.A/B测试及线上指标（转化率、任务完成率等）。
4. 长任务场景里，怎么避免Agent 无限推理或者死循环
  * **步数/时间限制**：硬性最大步数或超时阈值，超出则强制终止或请求人类介入。
  * **循环检测**：监控状态重复（动作、观察内容相似度），当检测到高度重复时触发重规划或退出。
  * **进展评估**：维护子目标检查点，若无进展则停止。
  * **终止条件提示**：在系统提示中明确“当无法继续时输出FINISH并说明理由”。
  * **置信度阈值**：模型对动作附置信度，低置信且连续失败时主动停掉。
  * **人类在环**：设置可被回调的批准步骤，或自动标记可疑循环交给人类。
5. 一个agent loop的控制性体现在哪些方面?在模型侧能不能解决?
  * 控制性体现在：
    * **行为可预测**：给定相同输入和状态，产生一致的输出序列。
    * **权限边界**：可限制可用工具集、可访问资源、最大副作用。
    * **中断与恢复**：可随时暂停、检查状态、修改指令再继续。
    * **强制约束**：必须遵循的规则（如输出格式、禁止某些操作）。
    * **审计日志**：每一步可追溯。
  * 模型侧能解决一部分：通过训练（指令微调、RLHF）让模型学会遵循约束和输出格式，但仍无法保证绝对遵循，尤其对抗性输入或分布外情况。完全的控制需要工程手段（沙箱、解析器、状态机、权限校验）与模型能力叠加。
6. 支持长程任务的agent需要注意哪些问题(验证、上下文焦虑等)
  * **上下文窗口焦虑**：长任务产生大量历史，超出窗口则丢失信息。需要压缩、摘要、分层记忆。
  * **任务验证与自检**：需要中间步骤验证机制（如断言、测试、结果比对），及时纠偏。
  * **错误累积**：早期错误滚雪球，需频繁检查点并支持回滚。
  * **目标漂移**：长时间推理容易偏离原始意图，需不断对齐子目标与主目标。
  * **资源管理**：管理文件、环境状态，避免残留状态污染后续步骤。
  * **人类介入接口**：设置可插入审批和引导的时机，防止偏离太远。
  * **遗忘与记忆巩固**：将重要发现写入长期记忆，避免重复试错。
7. coding agent 你了解过哪些 bench
  * ⭐️**SWE-bench (Verified/Full)**：从真实GitHub Issue中提取，让Agent修Bug/实现特性，自动运行测试评估。
  * **HumanEval / MBPP**：函数级代码生成，但不完全适合Agent。
  * **DS-1000**：数据科学任务，涉及多库使用。
  * **CodeContests**：竞赛编程题。
  * **BIG-bench** 中的代码任务。
  * **WebArena / MiniWoB++**：部分任务涉及网页代码交互。
  * **InterCode**：交互式代码环境（命令行、SQL等），评估Agent执行及反馈循环。
  * **The Stack / RepoBench**：仓库级代码理解与补全。
  * **Terminal Bench**：命令行Agent基准。
8. 怎么设计一个coding agent 的评测 pipeline 架构
  * **任务定义层**：收集Issue、规范描述、预期修改和测试用例。
  * **环境隔离层**：为每个评测实例启动容器或沙箱，提供一致的代码库快照。
  * **Agent交互层**：通过API与模型交互，Agent可执行命令、编辑文件、运行测试。需捕获stdout、stderr、文件变更。
  * **反馈日志记录**：记录每一步的工具调用、输出、推理过程、时间戳。
  * **评估器**：自动应用测试套件，比对期望的修改（Diff、测试通过率）。支持非确定性多次运行。
  * **评分与统计**：计算Pass@k，平均步数、Token消耗、成功率等。
  * **可复现与扩展**：Pipeline应基于Docker，支持并行批量评估，结果存入数据库，便于分析错误模式。
9. coding agent需要收集的数据有哪些
  * **任务描述**：Issue文本、需求说明。
  * **环境状态**：初始仓库代码、依赖信息。
  * **Agent轨迹**：每一步的思考/计划、工具调用（命令、编辑操作）、工具输出。
  * **人类修正记录（若有人类介入）**：编辑的补丁、命令纠正。
  * **最终结果**：代码补丁、测试通过/失败详情。
  * **元数据**：模型标识、耗时、Token消耗、硬件环境。
10. Kimi-Swarm?deep research?
  * Kimi-Swarm：月之暗面（Moonshot AI）提出的智能体框架，核心是用“群体”方式组织多个Kimi Agent，通过共享消息池和动态路由协作解决复杂任务。亮点是去中心化的协作与上下文管理，支持并行任务分发和结果聚合，适合长文档解析、多步调研等。
  * Deep Research：通常指OpenAI的“深度研究”功能（或Google的类似功能），Agent可以自主进行多步网络搜索、阅读、综合信息，最终生成带有引用的研究报告。其核心是长程规划、多源信息交叉验证、自我提问深化，以及超长上下文（可能数百万Token）下的记忆整合。
11. 主流框架Claude Code、Codex、OpenClaw
  * **Claude Code**：Anthropic推出的终端Agent，可直接在项目仓库中理解代码、执行命令、编辑文件。特点是紧密集成Claude模型，强调安全、权限控制和逐步确认。采用Agent Loop模式，有工具集（Bash、Edit、Write等），具备记忆和规划能力。
  * **Codex**：OpenAI的Codex（通常指基于GPT的代码模型及其API，或Codex CLI）。早期Codex强调代码补全，现在可指Codex CLI/OAI的Agent产品。允许在终端执行多步编辑、测试、调试，与GitHub集成。设计上偏重快速迭代与代码生成，工具调用灵活。
  * **OpenClaw**：开源社区的多模态桌面Agent框架，主要面向计算机使用（computer use），可操控鼠标键盘、识别屏幕。强调跨应用自动化，架构上包含视觉理解+动作执行，通常与GUI Agent基准（如OSWorld）关联。它的Agent Loop集成视觉、规划和细粒度操作。

### Skill

1. skil直接塞进system prompt里，如果skil太多，占用上下文窗口太大，怎么处理
  * **按需加载**：仅将技能名称和简短描述放入system prompt，详细指令在执行时通过工具调用（如“加载技能详情”）动态注入到当前上下文。
  * **分块检索**：将技能描述向量化，根据当前任务意图检索最相关的Top-K技能，拼入提示。
  * **分层提示**：在system prompt放置技能索引，明确“当需要技能X时，调用use_skill(X)获取完整说明”，由Agent主动拉取。
  * **压缩描述**：用精炼的关键词、触发条件代替长文本，并在需要时展开。
  * **微调模型内化技能**：将高频技能通过微调植入模型权重，减少提示依赖。
2. 如果两个skill实际内容不一样，但是description很相似，导致agent每次加载错skill，怎么解决
  * **增加区分度**：修改描述，突出关键差异（如前置条件、输出格式、限制场景）。
  * **添加入参示例**：在描述中附上典型使用场景和误用反例，使模型能依据上下文区分。
  * **双阶段选择**：第一层粗选候选技能，第二层在加载详细指令后让模型重新判断并确认或切换。
  * **强制澄清**：若模型不确定，提示其向用户询问或输出需要澄清的标志，由外部逻辑指派。
  * **使用更丰富的嵌入检索**：如果基于检索选择技能，可对完整技能文档进行嵌入匹配，而非仅靠简短描述。
  * **追踪和纠正**：记录加载错的情况，通过微调或提示优化来修正。
3. Skill怎么加载卸载
  * **加载**：Agent依据任务决定需要某个技能时，通过调用内部工具（如load_skill(name)），该工具从技能库取出完整指令/代码/资源，注入到当前对话上下文或作为系统消息追加。也可用动态模板，将技能详情渲染进下一轮prompt。
  * **卸载**：当Agent输出unload_skill(name)或任务上下文切换时，从活跃上下文移除技能详情，释放Token。实现上，需要维护当前激活技能列表，支持移除并重新组装系统提示。
  * **上下文分区**：使用独立的信息区（如平行系统消息）存放技能，支持替换和删除而不干扰主对话。
4. 怎么存储更新
  * **存储**：技能可存为结构化数据（YAML/JSON），包含名称、描述、触发条件、提示模板、关联工具、代码包等。可存储在数据库、文件系统或向量数据库中。
  * **版本控制**：使用Git管理技能变更，每个技能有版本号，Agent可请求特定版本。
  * **更新流程**：通过管理后台编辑、发布，或由Agent自我改进产出新技能（需审核）。更新后可广播给运行中的实例，或在下次加载时生效。使用CI/CD测试技能效果再上线。
5. 背后的sandbox怎么实现
  * **容器沙箱**：每个技能执行环境用Docker/gVisor/Firecracker隔离，限制网络、文件系统和系统调用。
  * **WebAssembly**：轻量级，跨平台，适合运行小段逻辑。
  * **进程级隔离**：使用seccomp、cgroups等限制资源与系统调用，配合临时文件系统。
  * **Python沙箱**：如RestrictedPython、Pyodide，禁用危险模块。
  * **语言解释器**：对代码类技能，使用受控解释器执行并捕获输出。
  * **资源限制**：设定CPU、内存、超时、无网络或白名单域名访问。
6. skill怎么评估
  * **单元测试**：针对技能的输入输出编写测试用例，验证其正确性。
  * **成功率**：在模拟任务中，Agent使用该技能完成子任务的准确率。
  * **效率提升**：对比有无该技能时完成任务的总步数、时间和Token消耗。
  * **冲突检测**：与其他技能共存时是否引起选择混淆或错误。
  * **人工评审**：技能描述清晰度、安全性、边界条件。
  * **线上A/B测试**：放量观察对任务完成率和用户体验的影响。

### Context

1. 上下文工程有哪些要注意的点
  * **信息密度**：避免冗余，优先保留与当前任务相关的信息。
  * **结构清晰**：使用XML或Markdown分节，使模型更容易聚焦。
  * **动态裁剪与摘要**：长历史需智能压缩，保留关键决策和结果。
  * **权限与污染**：防止历史中不可信工具的返回信息污染指令。
  * **Token与成本优化**：平衡上下文长度与推理质量。
  * **版本控制和模板化**：提示模板管理，方便迭代。
  * **上下文注入安全**：防Prompt Injection。
2. Prompt Injection在Agent场景里怎么防
  * **输入净化与隔离**：对外部数据（网页、邮件内容）用特定分隔符包裹，并明确告知模型“以下内容不可信任”。
  * **权限最小化**：工具应有严格权限限制，敏感操作（如删除、付款）需用户二次确认，且不盲目信任模型输出。
  * **结构化输出解析**：不直接执行模型输出的自然语言，而是通过解析确定的函数调用格式，过滤异常字段。
  * **二次审核**：在关键动作前，将拟定操作呈现给用户批准。
  * **多级提示架构**：使用系统提示与用户提示分离，将不可信数据仅放入用户消息且明确标注角色。
  * **模型训练增强抗注入**：通过对抗训练提高模型对指令越权的鲁棒性。
  * **上下文监测**：监控提示中是否出现典型注入模式（如“忽略之前指令”），触发告警或拒绝执行。
3. 压缩机制怎么实现的
  * **摘要压缩**：调用模型对历史分段进行摘要，替换原文。
  * **嵌入+检索压缩**：将历史转成嵌入，根据当前查询检索最相关的片段注入。
  * **关键信息提取**：提取实体、决策、错误等关键因子，抛弃细节叙述。
  * **滑动窗口**：保留最近的K轮完整对话，更早的仅保留摘要或丢弃。
  * **结构化内存写回**：将重要结论写入外部存储，仅保留引用。
  * **迭代压缩**：当上下文将满时，对早期部分压缩后重新拼接。
  * **模型自压缩**：如LLMLingua等方法，用小模型精炼提示中冗余Token。

### Memory

1. Agent的Memory一般怎么做
  * **短期记忆**：对话历史窗口，直接作为上下文。
  * **长期记忆（显式）**：外部存储（向量DB、传统DB、键值存储），将用户信息、偏好、事实等提取并存储，在需要时检索。
  * **工作记忆**：任务执行过程中的暂存区，如记事本、变量缓存，可利用工具读写。
  * **情景记忆**：记录过去成功/失败的完整轨迹，供未来类似场景参考。
  * **语义记忆**：结构化知识库或知识图谱。
2. 你怎么设计Agent的长期记忆写回策略、衰减策略和冲突消解
  * 写回策略：
    * 重要性触发：当信息被多次引用或标记为重要时才写入。
    * 时间/事件触发：会话结束时总结要点写入；或在检测到冲突、遗忘前写入。
    * 主动询问：“我该记住这件事吗？”经用户确认后写入。
  * 衰减策略：
    * 基于时间衰减权重，最近访问或更新的记忆权重高。
    * 基于访问频率，久未使用可归档，检索时降低排名。
    * 设置过期时间，如偏好类长期保留，临时上下文短期留存。
  * 冲突消解：
    * 当新记忆与旧记忆矛盾（如“住址：北京” vs “住址：上海”），需判断来源可信度、时间戳，新信息优先但需验证。
    * 采用多版本保存，检索时返回最新版本或标记冲突提醒Agent需澄清。
    * 更新时使用“软删除”旧值，保留历史便于恢复。
    * 对同一属性使用确定性合并规则（如列表去重合并，标量用最新）。
3. 如何量化评估memory机制的效果
  * **记忆召回率/精准率**：模拟问询已知信息，看是否正确召回。
  * **任务性能提升**：对比有记忆和无记忆时，长对话中任务完成度、重复提问次数、用户澄清次数。
  * **插入/更新延迟**：记忆写回操作耗时及对交互速度的影响。
  * **冲突解决准确率**：构造矛盾信息场景，看系统是否按预期消解或提示。
  * **遗忘曲线模拟**：间隔一段时间再问，检查记忆保留情况。
  * **用户研究**：满意度、信任度主观评价。
4. self-improve实现细节
  * **反思生成**：任务结束后，Agent分析失败原因，生成改进要点。
  * **经验存储**：将反思结果结构化存入长期记忆，如“执行X时若遇到Y则应Z”。
  * **检索增强**：未来类似任务时，检索相关经验注入提示，指导规划或工具选择。
  * **微调循环**：收集成功轨迹与人工修正，对模型进行SFT/DPO更新，使行为内化。
  * **动态提示优化**：通过元Agent评估提示效果并自动调整系统提示。
  * **技能生成**：将成功的执行序列封装成新技能（Skill），可被调用。
  * **验证与回滚**：所有自动生成的改进需经沙箱测试或人工审批，防止退化。

### Tools

1. 怎么做并行ToolCalling，既提升吞吐又保证一致性和可回放
  * **分析依赖**：解析工具调用间的数据依赖，生成有向无环图（DAG）。无依赖的可并行发出。
  * **预测与回滚**：保守执行，若并行调用间可能存在隐藏冲突，可以以事务方式执行：并行启动，若任一失败或结果导致不一致，则回滚（或使用补偿操作）。
  * **幂等性设计**：将工具设计为幂等，即使重复调用结果一致，方便重放。
  * **版本化环境**：记录每次并行调用的输入及产生的输出，结合环境快照，保证可以单步重放调试。
  * **顺序化依赖注入**：若B依赖A的输出，先将A的输出填充到B的参数后再调用；框架应自动串行化这些依赖链，其余并行。
  * **日志与Trace**：使用全局Trace ID记录每个Tool Call的精确顺序（逻辑时钟），即使物理并行也能还原顺序，便于重放。
2. Tool Calling一般怎么做，和普通函数调用有什么区别
  * Tool Calling流程：
    * 模型输出指定格式（JSON）包含函数名和参数。
    * 网关/执行器解析JSON，校验参数，执行对应函数。
    * 函数结果序列化后作为观察返回给模型。
  * 与普通函数调用区别：
    * 解释与生成：调用由模型从自然语言中推理生成，而非代码逻辑直接决定。
    * 非确定性：模型可能选择错误的工具或参数，需要验证和异常处理。
    * 异步与网络：通常为远程API，涉及网络开销和鉴权。
    * 安全风险：需严格沙箱和参数过滤，防止注入。
    * 可见性与控制：执行前往往需要审批层或人工确认，而非自动调用。
3. 如果线上发现Agent经常选错工具，你怎么排查
  * **日志分析**：抽取错误调用时的对话上下文、模型输出、可用工具列表，检查是否描述歧义。
  * **工具描述优化**：对比正确调用与错误调用的场景，微调工具名称、描述，增加区别度或添加示例。
  * **相似度检测**：计算工具间描述嵌入相似度，过高的须重新设计。
  * **模型能力评估**：测试同一模型在基准工具选择任务上的准确率；考虑换用推理更强的模型。
  * **上下文干扰**：是否历史中不当信息诱导了错误选择，检查上文Prompt Injection或误导内容。
  * **参数格式问题**：工具对参数格式要求严格，模型生成常出错，可在解析层增加容错和标准化。
  * **增加Few-shot示例**：在系统提示中加入正确使用工具的范例。
  * **边界情况收集**：积累Badcase，用于微调或规则修正。


### MCP

1. MCP、A2A、普通Function Calling 三者有么本质区别，工程里怎么选
  * 普通Function Calling：由模型提供商定义格式（如OpenAI的工具调用），模型输出调用指令，由用户侧代码执行并返回结果。是一种**模型内能力**，工具定义和生命周期由客户端管理。
  * MCP (Model Context Protocol)：Anthropic提出的开放协议，标准化客户端与工具服务器之间的通信。一个MCP服务器可以暴露资源、工具、提示模板等，模型通过MCP客户端发现并调用。核心是**工具生态的标准化和复用**，允许第三方开发工具服务器，Agent即插即用。
  * A2A (Agent-to-Agent)：Google提出的Agent间协作协议，定义Agent如何发现能力、协商任务、传递消息。与工具调用不同，A2A面向**对等Agent协作**，而非工具的单向调用。
  * 工程选型：
    * 需要单个Agent调用自定义内部API或函数：直接用Function Calling，简单高效。
    * 需要集成多个外部工具/数据源，希望工具标准化、易扩展：引入MCP，可复用社区工具服务器，解耦工具实现。
    * 多Agent系统，需跨智能体任务委派和通信：采用A2A。
  * 三者可组合使用，如Agent内部通过MCP连接工具，同时通过A2A与其他Agent对话。

### RAG

1. RAG接到Agent里，一般怎么设计
  * **工具化检索**：定义search_knowledge(query)工具，Agent需要背景知识时主动调用，检索结果以观察形式返回。可基于向量库、搜索引擎。
  * **透明注入**：在每轮自动根据用户问题检索相关文档，拼接在系统提示或用户消息中，Agent无感。适合简单场景，但缺乏自主控制。
  * **分层记忆**：短期用上下文，长期用RAG。Agent决策时检索长期记忆库补充上下文。
  * **多步推理RAG**：Agent可多次检索，组合不同查询深入获取信息，甚至对检索结果再提问。
  * **缓存与引用**：检索结果附带来源，Agent可在回答中引用，增强可信度。
  * **用户上下文增强**：将用户画像、历史对话片段存入向量库，动态检索丰富个性回复。

### Sandbox

1. 有了解Anthropic的managed agents吗?
  * 👉[原文](https://www.anthropic.com/engineering/managed-agents){:target="_blank"}👈
  * Anthropic的Managed Agents（如Claude Computer Use中的托管执行环境）是指由Anthropic管理的安全沙箱代理服务。主要特点：
    * 提供预配置的虚拟机或容器环境，运行Claude模型指定的桌面自动化动作（鼠标、键盘、截图）。
    * 沙箱内置安全策略，限制网络、敏感操作，自动重置环境防止状态污染。
    * 适用于需要计算机操作的Agent，免去用户自己搭建安全沙箱的复杂工作。
    * 重点解决安全性和可信执行，但灵活性受控于管理方。
2. 设计一个个人独占的sandbox架构,怎么承接toC大量的用户?会面临哪些问题
  * 架构：
    * **网关层**：用户请求接入，鉴权、分配沙箱实例。
    * **调度器**：管理预热或冷启动的沙箱池，基于负载动态扩缩。
    * **沙箱池**：为每个活跃用户保留一个独占容器/VM，通过快照和卷持久化用户个人数据。
    * **数据层**：用户主目录挂载持久化存储（如云盘），以便沙箱销毁后数据不丢失。
    * **生命周期管理**：闲置超时后自动挂起（保存状态并释放计算资源），用户回归时快速恢复。
  * 面临问题：
    * **资源成本**：为大量用户维持独占环境资源消耗大，需要高效的挂起/恢复和共享内核技术（如MicroVM）。
    * **冷启动延迟**：从零创建环境慢，需预热池或快速克隆。
    * **安全隔离**：必须严格隔离用户间、用户与宿主机，防止逃逸。
    * **数据持久化与备份**：保障用户文件不丢失。
    * **网络策略**：允许/限制出站访问，防止滥用。
    * **状态管理复杂性**：软件版本、用户自定义配置，需像管理个人电脑一样处理。
3. MicroVM这类虚拟化技术背后的实现
  * MicroVM（如Firecracker、QEMU microvm）是精简虚拟机，专为无服务器和容器化负载设计
  * 基于KVM：利用Linux内核的KVM实现硬件辅助虚拟化，每个MicroVM是一个轻量级VMM进程。
  * 最小化设备模拟：只提供极简的virtio设备（网卡、块存储），去除传统QEMU的复杂BIOS和PCI总线，启动时间<100ms。
  * 内存与CPU隔离：每个MicroVM有独立内核、内存空间，安全边界强于容器。
  * 精简内核：Guest内核通常裁剪掉不必要驱动，减小攻击面。
  * 快照与恢复：可对内存和寄存器做快照，实现快速挂起/恢复及克隆。
  * 与容器编排结合：如Firecracker被用在AWS Lambda和Fargate，每个函数在一个MicroVM中运行，兼具安全与密度。

### 推理优化

1. Prefil&Decode
  * Prefill（预填充）：一次性将全部输入提示送入模型，并行计算所有输入Token的Key-Value缓存（KV Cache）。这个阶段计算量大，但可高度并行。
  * Decode（解码）：逐Token自回归生成输出，每生成一个Token需利用所有历史KV Cache，访存密集、无法并行，是推理延迟的主要瓶颈。
2. 有哪些优化方式
  * **KV Cache优化**：PagedAttention（vLLM）将缓存分页以减少碎片；量化KV缓存（INT8/FP8）。
  * **连续批处理（Continuous Batching）**：动态组合新请求与正在生成的序列，提高GPU利用率。
  * **量化**：模型权重量化（GPTQ、AWQ）、激活量化降低显存和带宽。
  * **投机解码（Speculative Decoding）**：用小模型快速生成候选Token，大模型并行验证。
  * **算子融合**：合并计算层（如FlashAttention）减少显存读写。
  * **并行策略**：张量并行、流水线并行跨多GPU推理。
  * **预测性Prefill调度**：将Prefill与Decode分离部署（Disaggregated），独立扩展。
  * **前缀缓存**：相同系统提示或共享前缀，复用KV Cache。

### 模型

1. 后训练可以做什么
  * **指令微调（SFT）**：让模型更好地遵循指令、适应特定格式和任务。
  * **偏好对齐（RLHF/DPO）**：让输出更符合人类偏好，安全、有帮助、诚实。
  * **工具使用训练**：构造工具调用轨迹数据微调，提升函数选择与参数生成能力。
  * **长上下文扩展训练**：通过位置编码插值或继续预训练，延展上下文窗口。
  * **推理能力增强（如O1类的CoT内化）**：训练产生内部推理链，提高复杂任务表现。
  * **Agent专项训练**：使用Agent轨迹（思考-动作-观察）进行多轮微调，让模型适应Agent循环。
  * **安全与拒答训练**：增强对注入、危险指令的识别与拒绝能力。
  * **多语言与领域适应**：垂直领域数据继续训练，改善专业表现。

### Harness

1. 对harness的理解
  * Harness通常指模型评估框架，如EleutherAI的lm-evaluation-harness。它提供标准化流程：定义任务（数据集+指标），模型加载，生成或计算困惑度，输出度量结果。对Agent而言，Harness可扩展到Agent评估，集成沙箱、工具仿真和轨迹评估，成为可重复、可扩展的Agent评测流水线。
2. 从harness角度讲下你对上面几个框架分别的看法
  * Claude Code：因为其封闭生态，Harness可设计为通过终端模拟器与Claude Code交互，发送任务描述，捕获命令与文件变化，利用Docker提供隔离环境，以SWE-bench等测试用例自动评分。难点在于权限确认需模拟或旁路。
  * Codex (CLI)：类似Claude Code，但可能API更易编程调用。Harness可通过定义workspace、注入任务、收集代码diff和测试结果来评测。其工具调用接口标准化，使集成较方便。
  * OpenClaw：面向GUI Agent，Harness需配备桌面仿真环境（如OSWorld的虚拟机），捕获截图序列、鼠标键盘事件，评估任务完成状态（如UI元素匹配）。需处理视觉感知和动作空间，评测更复杂。
  * Harness的核心是把不同框架抽象成统一的Agent接口（感知、行动），然后插入标准任务和评估逻辑，实现公平对比。
