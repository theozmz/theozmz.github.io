## Experience

这是本人第一个完整的纯vibe coding工程实践项目。项目共持续5天，Harness CLI 用到 Claude Code，模型用的deepseek，一共烧了10亿tokens。


### AI Workflow

As we all know, Claude Code offers two modes: plan and edit. Firstly, I use the rough proposal from the early stage to generate the general functionalities directly with plan, then edit the complete code framework, and then refine it according to the functionalities. During the process, new functionalities are added, debugged, and unit tested. Finally, plan writes the documentation and conducts integration testing.

众所周知，Claude Code提供两种模式：plan和edit。首先，我把前期粗略的proposal直接用plan生成大致功能点，然后edit完整代码框架，然后按功能点细化，过程中伴随新功能点加入、debug和单元测试，最后plan撰写文档和集成测试。

### Claude Code + MCPs

Three MCPs were primarily utilized:

1. **UnityMCP**: The primary MCP used to connect Unity and Claude Code, allowing the creation of Assets, Prefabs, Scripts, and testing. Testing mainly involves compilation and play mode debugging.
2. **PiskelMCP**: Draw pixel art. **One interesting point is that although deepseek does not have multimodal capabilities, it can still read images with the support of cc.** Therefore, all our image materials are drawn using PiskelMCP, as well as all UI, special effects, and backgrounds, so there is no need to worry about copyright issues (laughs).
3. **FamiStudioMCP**: FamiStudio is an open-source sound synthesis software, and its native version does not support integration with MCP. **Therefore, I developed a MCP** myself, which is used to connect FamiStudio and Claude Code. It handles all the background music and sound effects in the game.


主要用到了3个MCPs：

1. **UnityMCP**: 蕞主要的MCP，用于连接Unity和Claude Code，可以创建Asset、Prefab、Scripts和测试，测试主要是编译和play mode调试。
2. **PiskelMCP**: 画像素画。**有趣的一点是，虽然deepseek不具备多模态能力，在cc的加持下它还是能读图。**于是我们的形象素材都是使用PiskelMCP画出来的，此外还有所有的UI、特效、背景，完全不用担心版权问题（笑。
3. **FamiStudioMCP**: FamiStudio是一个开源的声音合成软件，其原生版本并不支持接入MCP，**于是我自己开发了一个MCP**，用于连接FamiStudio和Claude Code。游戏所有的背景音乐和音效都是用这个生成的。

### Details

* **A bug that never gets fixed**
  * When encountering stubborn bugs, my advice is not to repeatedly ask the CLI to make changes at the same point while the edit mode is on. On the one hand, the AI may focus on a small part of the files in a certain local area or associated trace, piling up redundant code, which is not only useless but may also affect other logic. On the other hand, the generated redundant logic may be dead code in the early stage, but it may come back to life after adding new functional points later on, which leads to a series of bugs (we call this a feature, laugh). The solution is to first use the plan mode to let the AI conduct a comprehensive investigation of a problematic functional point, and then make modifications based on the results.
* **cc's phase even provides you with detailed options**
  * Let's plan a more complex feature for cc. I remember it involves turning scriptable objects into prefabs. Since there are many assets involved in the modification, and the script logic also needs to be changed accordingly, cc provides three options for each phase: 1. Test while modifying 2. Full change 3 give up This is quite impressive. However, it seems that this was the only occurrence during the development process.
* **Do it yourself or use MCP**
  Some people believe that the more MCP, the better, and even advocate encapsulating CLI runtime hooks, namely those bash scripts, into MCP logic. However, during development, cc, without certain functions (such as enabling deepseek to view images, where cc's solution is to write a python script to read binary files and dump them into plain text, which is even comprehensible, but horrifying), will start from setting up the environment and create the entire universe using python.
* **Don't rely too much on plan mode**
  * In the planning mode, personally, I feel that it's just about reading more documents and integrating comprehensive information. As for the next action mode based on these contexts, it largely depends on the action capabilities of the base model. Based on my experience with deepseek, it has most of the basic functions, and even supplements some points you might not expect, but it can't reach the finer details. When generating store weapon function points, cc and ds provide all the functions that can be run, but upon closer inspection of the textures, it's found that only assets are created without any drawings...
* **Can memories expire?**
  * Yes, during the development process, I noticed that cc would occasionally access a certain markdown file it had created, but I'm not sure about the specific updates.
* **The impact of auto-compact on code quality**
  * Auto-compact is the last and most reliable compression method among the five context compression methods in cc. It summarizes all contexts, consumes the most tokens, has the slowest speed, and suffers the greatest information loss, but it achieves the best compression effect.
  * If triggered during the planning process: It did not have a significant impact on the body's perception, suggesting that it is similar to a certain summarization method.
  * If triggered during the editing process: It doesn't seem to have caused a significant impact, as cc appears to retain some recent conversations. Although the underlying process is unknown, it appears that a task will abruptly end and be completed suddenly.
* **The rules manually added in CLAUDE.md may not be executed every time**
  * If I ask cc to update something or do something else at the beginning of the document (such as maintaining a dev log), sometimes it doesn't strictly follow through. Although there is a trade-off between the amount of code changes and updating the document, it has been observed that sometimes one or two small changes are recorded, and sometimes a whole large feature point is written but only output to the cmd, resulting in the document not being updated.


* **总也改不好的bug**
  * 遇到顽固的bug，我的建议是不要开着edit模式重复地要求CLI在某个相同的点上改。一方面AI可能会盯着某个局部或连带的trace上的一小部分文件，堆一些冗余的代码，既没用还有可能影响其他逻辑。另一方面，生成的冗余逻辑可能在前期是死代码，但是在后续加入新的功能点后可能重新复活，这就牵连出了一系列的bug（我们管这叫特性，笑）。解决办法是先用plan模式，让AI对某个出问题的功能点全面调查，然后根据结果进行修改。
* **cc的phase甚至会给你详细的选项**
  * 让cc plan一个比较复杂的功能点，我记得是把scriptable objects变成prefabs，由于涉及改动的素材很多，还有脚本逻辑也要跟着改，所以cc对每个phase都给了3个选项：1. 边改边测试 2. 全量更改 3. 放弃。这点比较惊艳。但是貌似开发的过程中，只出现了这一次。
* **自己整还是MCP**
  * 一些人的意见是MCP越多越好，甚至把CLI的runtime hook，也就是那些bash也封装进MCP逻辑中。但是在开发过程中，cc在没有一些功能（比如让deepseek看图，cc给出的解决方案是写python脚本读二进制文件dump成纯文本，甚至是可以理解的，恐怖如斯）的情况下，会从装环境开始，用python搓出整个宇宙。
* **别太指望plan mode**
  * 计划模式个人体感上觉得，只是看了更多的文件，整合了全面的信息。至于有了这些上下文，下一步的行动模式，更多的取决于base model的行动能力。就deepseek体验下来，大体上的东西该有的都有，甚至会补一些你想不到的点，但是更细节的地方触及不到。在生成商店武器功能点时，cc和ds给出了能跑的全部功能，但是细看贴图会发现只创建了asset没有画。。。
* **记忆会过期吗？**
  * 会。开发过程中，我发现cc偶尔会去看它自己创建的某个记忆markdown，具体更新没有不得而知。
* **auto-compact对代码质量的影响**
  * auto-compact是cc5种上下文压缩方式中，最后一层也是保底的压缩方式，对全部上下文全量摘要，消耗token最多，速度最慢，信息损失最大，但是压缩效果最好。
  * 如果在plan过程中触发：体感上并未造成巨大影响，推测这与某种摘要方法是相似的。
  * 如果在edit过程中触发：体感上同样未造成巨大影响，因为cc貌似会保留最近的一些对话。虽然背后的过程不得而知，但是看上去一个任务会戛然而止，突然完成那种。
* **CLAUDE.md中人为添加的规则不一定每次都执行**
  * 如果在文档开头让cc每次都更新一些什么或做什么其他的（比如我让它额外维护一个dev log），有些时候其并不会严格遵循。虽然会权衡一次代码更改量来更新文档，但是观察到有时一两处小改动也记录，有时一整个大功能点写完却只是输出到cmd，导致文档没有更新。

## Proposal

### Motivation & Inspiration

#### Objective

We are driven by a desire to shift strategic depth from pre-defined loadouts to moment-to-moment creativity in fast-paced combat. *Spellweaver: The Element Mage* challenges players to combine elemental fragments on the fly, creating spells that adapt to the flow of battle. This "pick-up and cast" system rewards improvisation and mastery.

- Dead Cells: Its fluid 2D action and death-as-progression loop define our combat feel.
- Noita: The deep, systemic spell and element interaction inspires our fusion mechanics.
- The Binding of Isaac: Rich item synergies and layered endings inform our dual-conclusion structure.
- Celeste: Its outstanding jumping system inspires our action part.
- The Spell Brigade: The concept of spell crafting through active combination grounds our elemental slot system.

#### Concept & Perspective

##### What is it?

*Spellweaver: The Element Mage* is a 2D action-roguelike where players combine elemental fragments in real time to forge spells. Set in a decaying underground civilization, you must fight through enemies corrupted by the old world's mistakes and decide whether to preserve a broken order or risk everything for renewal.

##### Concept

The player controls a cloned "Elemental Weaver" escaping a collapsing underground. Combat revolves around three spell slots, each holding a base element (fire, ice, lightning). By collecting elemental fragments mid-battle, players instantly create combo spells. Each fragment is one-use, forcing constant adaptation. The game features two endings based on hidden "truth fragments" collected throughout runs.

##### Perspective

Side-scrolling 2D action with pixel art visuals, viewed from a third-person side-on perspective.

### Idea

#### World

A once-great magical civilization lies buried beneath a surface sealed by the "Silent Storm." The last survivors, the "Order of Equilibrium," maintain a fragile underground sanctuary by siphoning power from the planet’s core—a process that slowly kills the world. Players navigate ruins, prisons, and geothermal reactors, uncovering echoes of the old world and the sins of the new.

#### Story

You are a clone of an ancient Elemental Weaver, created to wield powers that modern mages fear. As you fight through the Order's defenses, you face puppets of former heroes and learn that your existence was engineered to either replace the dying ruler or perpetuate the Order's control. Your final choice determines whether the world decays or is reborn.

#### Characters

- The Protagonist: A cloned Weaver, searching for identity and freedom.
- Valcary, the Executioner: A lobotomized valkyrie and the player's former trainer, serving as the tutorial boss.
- Hephaestus, the Grand Artificer: The Order's engineer, who sees sacrifice as necessity.
- Moira, the Memory Weaver: A fractured oracle who reveals the cyclical nature of the world.
- Augustus, the Imperishable: The final ruler, whose true form is bound to the dying core.

![characters](../../../../images/posts/2026/05/game/characters.png)

#### Core Actions

- Movement: WASD for movement, Space to jump, Shift to dash.
- Combat: Mouse-aimed shooting (reload with R) combined with instant spellcasting via number keys 1–3.
- Spellcrafting: Each spell slot holds a base element. Collecting a fragment (e.g., Burst, Chain, Pierce) triggers a new spell. The slot glows to indicate a new spell is ready.
- Environmental Interaction: Element-locked chests require attacking three elemental symbols in sequence using matching spells.

#### Core Loop

Explore procedurally generated rooms → defeat enemies → collect elemental fragments and upgrades → craft and cast spells in combat → unlock new paths and lore → face bosses → uncover endings based on truth fragments collected.

#### UI Journey

- Main Hub: Spell slots, health, and mana displayed in the lower corners.
- Combat: Fragments collected trigger visual and audio feedback; spell slots glow when a combo is ready.
- Element Lock: On-screen sequence display prompts timed spell attacks.
- Between Runs: Progression tracked through unlockable elements and truth fragment collection.

#### Motivation Flow

Players are motivated by mastery of the dynamic spell system, the tension of high-stakes combat, and narrative discovery. Each run offers new fragment combinations, encouraging experimentation. The dual-ending system rewards thorough exploration, while lore fragments reveal the tragic history of the world and its characters.

![flow](../../../../images/posts/2026/05/game/flow.png)

### Four Basic Elements

#### Mechanics

Real-time action with a unique spellcrafting loop: three spell slots, base elements, and single-use fragments enable emergent combat strategies. Procedural rooms and element-locked puzzles add variety. Dual endings based on collectible truth fragments create replayability.

#### Story

A morally complex narrative set in a decaying magical civilization. The player uncovers their own artificial origin and decides between preserving a dying order or risking renewal. Enemies are former heroes corrupted by the system, adding emotional weight to encounters.

#### Aesthetics

Pixel art with dynamic lighting and particle-driven spell effects. The visual style contrasts cold industrial ruins with vibrant magical energy. UI design echoes ancient grimoires and elemental glyphs.

#### Technology  

Built in Unity 2022 LTS with C#. Core systems include a data-driven spell combination manager, tilemap-based procedural generation, and state-machine-driven enemy AI.

### Plan

#### Methodologies

We will follow an iterative development process, beginning with core mechanics, layering procedural systems, and finishing with content integration and polish. Weekly milestones will track progress across three parallel tracks: combat mechanics, level systems, and art production.

#### Platform

Unity 2022 LTS or Tuanjie Engine, Visual Studio Code, GitHub, and WeChat or Zoom for task management.
